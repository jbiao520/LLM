# 长上下文（Long Context）科普版

> 面向零基础读者的长上下文技术入门指南

## 一句话概括

**长上下文技术让模型能"记住"更多内容，从读短文���成读整本书。**

## 从一个问题开始

你有没有遇到过这种情况：
- 让 ChatGPT 总结一篇长文章，它说"太长了"
- 聊着聊着，模型忘了之前说的话
- 上传一个长文档，模型只看了开头

**这就是上下文长度限制的问题。**

## 什么是上下文长度？

### 类比：短时记忆

模型的上下文就像人的"短时记忆"：
- GPT-3：约 2000 字
- GPT-4：约 8000 字（标准版）/ 128000 字（长上下文版）
- Claude 3：约 200000 字

### 限制的原因

```
┌─────────────────────────────────────────────────────────────────┐
│                   为什么上下文有限制？                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. 计算成本                                                   │
│      注意力计算是 O(n²) —— 长度翻倍，计算量翻四倍！              │
│                                                                 │
│   2. 内存限制                                                   │

│                                                                 │
│   3. 训练时的限制                                               │
│      模型在训练时没见过那么长的序列                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 长上下文技术

### 1. RoPE 缩放

**核心思想**：调整位置编码的"尺度"

```
原始 RoPE:        缩放后:
位置 1,2,3...     位置 0.5, 1.0, 1.5...

就像把地图缩小，原本 1-100 的位置压缩到 1-50
腾出空间放更多内容
```

### 2. 线性缩放

最简单的方法：

```
原始: 位置 1, 2, 3, ..., 4096
缩放: 位置 0.5, 1.0, 1.5, ..., 2048

现在可以在 2048-4096 的位置放新内容
```

### 3. 动态 NTK 缩放

更智能的缩放方式，根据输入长度动态调整：

```
短输入: 不缩放
长输入: 逐渐增大缩放因子
```

### 4. YaRN（Yet another RoPE extensioN）

结合多种技术，效果最好：

```
1. 温度缩放
2. 厚度缩放
3. 动态调整

可以把 4K 上下文扩展到 128K+
```

## 实际应用场景

| 场景 | 所需上下文 |
|------|-----------|
| 日常对话 | 4K 足够 |
| 文档问答 | 32K+ |
| 代码分析 | 32K-128K |
| 长篇小说总结 | 128K+ |
| 多文档分析 | 200K+ |

## 挑战与权衡

### 精度下降

扩展太远会导致模型"迷糊"：

```
原始训练: 4K
扩展到: 128K (32倍)

模型可能在长文本末尾表现下降
```

### 计算成本

更长的上下文 = 更高的成本：

```
4K 上下文: $X
128K 上下文: ~$32X (理论值，实际优化后会低一些)
```

## 总结

| YaRN | 最先进的长上下文技术 |
