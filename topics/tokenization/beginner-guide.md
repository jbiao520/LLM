# 分词器（Tokenization）科普版

> 面向零基础读者的分词器入门指南

## 一句话概括

**分词器是文本和模型之间的"翻译官"，把人类文字变成模型能理解的数字。**

## 从一个问题开始

当你和 ChatGPT 对话时，你输入的是文字，但模型实际上只能处理数字。

那么问题来了：**怎么把 "你好世界" 变成数字？**

最简单的方法是给每个字符一个编号：
- "你" = 1
- "好" = 2
- "世" = 3
- "界" = 4

但这样有问题：
- 字符太多了（中文有数万个字）
- 很多词组合在一起才有意义（如"人工智能"）

**分词器就是解决这个问题的智能"翻译官"。**

## 什么是 Token（词元）？

Token 是分词器处理的基本单位，它可以是：

| 类型 | 示例 | 特点 |
|------|------|------|
| 词级别 | "人工智能" = 一个 token | 语义完整，但词表巨大 |
| 字符级别 | "人工智能" = 四个 token | 词表小，但语义被切分 |
| 子词级别 | "人工智能" = "人工" + "智能" | 平衡语义和词表大小 |

**现代 LLM 普遍使用子词级别**，因为它：
- 词表大小适中（通常 3万-10万）
- 能处理生僻词和新词
- 保留一定的语义信息

## BPE：最常用的分词算法

### 类比：压缩文件的智慧

想象你要压缩一本书，你会怎么做？

1. 找出最常见的字组合
2. 给常见组合一个缩写
3. 用缩写替换原文

**BPE（Byte Pair Encoding）就是这样工作的：**

```
原始文本：低 低 中 中 高 高 高

第1轮：发现 "高 高" 最常见，合并为 "高高"
结果：低 低 中 中 高高 高

第2轮：发现 "低 低" 最常见，合并为 "低低"
结果：低低 中 中 高高 高

...继续合并...
```

### BPE 的工作流程

```
┌─────────────────────────────────────────────────────┐
│  输入文本："我喜欢学习人工智能"                        │
└─────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│  第一步：拆分为字符                                    │
│  ["我", "喜", "欢", "学", "习", "人", "工", "智", "能"] │
└─────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│  第二步：查找词表，合并已知词元                          │
│  ["我", "喜欢", "学习", "人工智能"]                     │
└─────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│  第三步：转换为 ID                                     │
│  [9527, 3821, 4523, 7892]                            │
└─────────────────────────────────────────────────────┘
```

## 为什么分词很重要？

### 1. 影响模型理解

同样的词，分词方式不同会导致模型理解不同：

```
英文例子：
"don't" 可能被分成：
- ["don", "'", "t"]  ← 不好的分词
- ["don", "'t"]      ← 更好的分词
- ["do", "n't"]      ← 另一种方式
```

### 2. 影响 API 成本

大多数 LLM API 按 token 数量计费：

| 文本 | 粗略 Token 数 |
|------|--------------|
| 1 个英文单词 | ~1 token |
| 1 个中文字 | ~1-2 tokens |
| 1000 个英文字符 | ~250-300 tokens |

**优化 prompt 可以节省成本！**

### 3. 影响模型行为

分词器的"怪癖"会影响模型输出：

```
为什么模型有时会在单词中间加空格？
→ 因为那个词被分成了多个 token

为什么有些数字计算会出错？
→ 因为数字被奇怪地分割了（如 "12345" → ["12", "345"]）
```

## 常见的分词器

| 分词器 | 使用模型 | 特点 |
|--------|---------|------|
| tiktoken | GPT-4, GPT-3.5 | OpenAI 官方，BPE 变体 |
| SentencePiece | LLaMA, Mistral | 支持多种算法，开源 |
| tokenizer | Claude | Anthropic 使用 |
| WordPiece | BERT | Google 开发 |

### tiktoken 示例（GPT 系列）

```python
import tiktoken

# 加载 GPT-4 的分词器
enc = tiktoken.encoding_for_model("gpt-4")

# 编码
text = "Hello, 世界！"
tokens = enc.encode(text)
print(tokens)  # [9906, 11, 16256, 244, 98320]

# 解码
decoded = enc.decode(tokens)
print(decoded)  # "Hello, 世界！"
```

## 特殊 Token

分词器通常有一些特殊的保留 token：

| Token | 作用 |
|-------|------|
| `<\|begin\|>` | 文本开始 |
| `<\|end\|>` | 文本结束 |
| `<\|pad\|>` | 填充短序列 |
| `<\|unk\|>` | 未知词 |
| `<\|sep\|>` | 分隔不同部分 |

这些 token 帮助模型理解文本结构。

## 分词器的局限性

### 1. 逆序词问题

```
"star" 和 "stars" 的 token 可能完全不同
导致模型难以理解它们是同一个词的不同形式
```

### 2. 数字问题

```
"12345" 可能被分成 ["123", "45"]
这会影响模型的数学能力
```

### 3. 代码问题

```
Python 的缩进有时会被奇怪地处理
影响模型生成代码的质量
```

## 直观对比

```
输入："The quick brown fox jumps over the lazy dog"

tiktoken (GPT-4):
["The", " quick", " brown", " fox", " jumps", " over", " the", " lazy", " dog"]
9 tokens

如果用字符级别：
84 tokens（每个字符一个）
```

**好的分词器能显著减少 token 数量，提高效率。**

## 总结

| 概念 | 通俗理解 |
|------|----------|
| Token | 文本被切分的最小单位 |
| 分词器 | 文本和数字之间的翻译官 |
| BPE | 最常用的分词算法，智能合并常见组合 |
| 词表 | 所有 token 和对应 ID 的字典 |
| 特殊 Token | 有特殊含义的保留 token |

## 实用技巧

1. **查看 token 数量**：使用 `tiktoken` 预估 API 成本
2. **优化 prompt**：简洁的表达可以减少 token
3. **注意边界**：某些字符组合可能产生意外的 token
